* PTY Investigation (virtme path)

** Problem

=vng -e <command>= runs without a PTY. Bash warns "no job control",
"cannot set terminal process group". Stdin redirection broken.
Long lines wrap incorrectly (terminal width unknown).

=vng= without =-e= has a proper PTY but doesn't run our setup.

** Attempted solutions

- =2> >(grep -v ...)= on bash: blocks startup (grep waits for EOF)
- =python3 pty.spawn(['vng', ...])=: PTY wraps host-side vng, not
  the guest-side bash. Doesn't help.
- =vng --shell /path/to/wrapper=: flag exists in =--help= but
  wrapper never executed. Source grep inconclusive — may be in
  a different module or generated dynamically.
- =PROMPT_COMMAND= export: doesn't propagate into VM (fresh kernel
  boot, not a subshell).
- =vng --user root= without =-e=: proper PTY but setup doesn't run.

** Current approach: researching =vng --shell=

Agent searching virtme-ng GitHub, docs, source for how =--shell=
actually works. May enable: =vng --user root --shell /wrapper= where
wrapper sources setup then execs bash with proper PTY.

* Current State (working, no PTY)

=guest-interactive.sh=:
- Creates ext4 + XFS loopback with quotas
- Creates =data/= dirs with chmod 777 (for =runuser -u nobody=)
- Prints banner with kernel info, mount paths, examples
- Notes about no-PTY limitations
- Drops to =bash --norc --noprofile -i=
- On exit: teardown message, fs-setup EXIT trap cleans up

=boot.sh=:
- =boot_kernel_interactive()= dispatches to virtme or QEMU
- =_boot_virtme_interactive()=: =vng -e guest-interactive.sh=
- =_boot_qemu_interactive()=: QEMU with =__INTERACTIVE__= marker,
  init drops to bash on serial console

=init= (initramfs):
- Detects =__INTERACTIVE__= in command
- Both 9p and rootfs modes: =chroot ... /bin/bash -l= instead of
  running command + poweroff
