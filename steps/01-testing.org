* Overview

Build a local-first, CI-agnostic test framework that tests quotatool
across 15-20+ Linux kernel versions (including vendor-patched kernels)
and two filesystem types (ext4, XFS).

Architecture:
#+begin_example
test/                          ← workspace (all test framework code)
  run-tests.sh                 ← entry point: "run one command, get results"
  lib/
    boot.sh                    ← kernel boot layer (virtme-ng / raw QEMU)
    fs-setup.sh                ← loopback filesystem create/mount/teardown
    tap.sh                     ← TAP output helpers
  tests/
    t-block-limits.sh          ← individual test scripts
    t-inode-limits.sh
    t-grace-period.sh
    ...
  kernels/
    kernels.conf               ← matrix definition (name, vmlinuz path, boot method)
    download.sh                ← fetch + extract vendor kernels
    initramfs/                 ← custom initramfs for legacy kernels
  results/                     ← output (gitignored)

.github/workflows/test.yml    ← thin CI wrapper (calls run-tests.sh)
.cirrus.yml                   ← FreeBSD testing
#+end_example

Background: =research/testing-infrastructure.org=,
            =research/multi-kernel-testing.org=,
            =research/quotatool-build-analysis.org=

Key finding: quotatool ships its own quota struct definitions
(=src/linux/linux_quota.h=, =xfs_quota.h=, =dqblk_old.h=, =dqblk_v0.h=)
and does runtime detection of the kernel's quota interface in
=src/linux/quota.c:kern_quota_format()=. *One binary works across all
kernel versions.* Build once, test everywhere. But the runtime detection
code is the critical fragile point — it was last updated around the 4.x
kernel era and likely needs fixes for modern kernels.

| U01  | Human-readable units (K,M,G,T)  | ext4      | parse    |
| U02  | Relative adjustment (+100M)     | ext4      | parse    |
| F01  | Raise-only flag (-R)            | ext4      | feature  |
| F02  | Grace period set (-t)           | ext4      | feature  |
| F03  | Grace period restart (-r)       | ext4      | feature  |
| F04  | Numeric UID (:12345)            | ext4      | feature  |
| O01  | Verbose output (-v)             | ext4      | output   |
| O02  | Dump/machine-readable (-d)      | ext4      | output   |
| E01  | No root → graceful failure      | ext4      | error    |
| E02  | Invalid arguments               | ext4      | error    |
| E03  | Non-existent mount point        | ext4      | error    |
| D01  | Format detection succeeds       | ext4      | detect   |
| D02  | Format detection succeeds       | XFS       | detect   |
| D03  | Get+set round-trip (set, get back, compare) | ext4, XFS | detect |

*Category "detect"*: These test =kern_quota_format()= — the runtime
format detection in =src/linux/quota.c=. This is the most fragile code
path and the primary reason for multi-kernel testing. It probes
=/proc/fs/quota=, =/proc/sys/fs/quota=, and falls back through
=Q_V0_GETSTATS= → =Q_OLD_GETSTATS= → =Q_OLD_GETQUOTA=. It contains
vendor-specific workarounds (e.g., RedHat 2.4.2-2) and has not been
updated since ~kernel 4.x. *Any kernel where detection fails = test
failure = investigation needed.* These tests are the early warning
system for the entire project.

/Verification/: test list reviewed, each test has clear pass criteria.

* Phase 6: Kernel Matrix Maintenance [0/1]

The kernel matrix is a snapshot. Distros release new kernels
continuously. Without a maintenance strategy, the matrix goes stale
within 12-24 months and the test framework loses its value.

** TODO 6.1 Future-proof the kernel matrix

Design a sustainable process for keeping =kernels.conf= current as
distros release new versions and old ones go EOL. This is not a
one-time task — it's an ongoing concern that needs a low-friction
solution.

Problems to solve:
- *Discovery*: How do we learn that Ubuntu 26.04, AlmaLinux 10,
  Fedora 44, Debian 14, etc. have been released? Manual tracking
  doesn't scale and is easy to forget.
- *URL sourcing*: Each distro has different repository layouts,
  package naming, and mirror structures. URLs break over time
  (packages move from =security.ubuntu.com= to =old-releases=,
  Fedora moves from =releases/= to =archive/=, etc.).
- *Tier rotation*: When a new LTS ships, the old one should move
  from tier 1 → tier 2 → tier 3 → eventually drop. Needs clear
  rules (e.g., "tier 1 = actively supported LTS/stable").
- *Validation*: New kernels may need new mkfs flags, new module
  deps, new boot quirks. Adding a kernel isn't just a URL —
  it needs a test run.
- *Minimum viable process*: Something a single maintainer can do
  in 30 minutes twice a year, not a full-time job.

Possible approaches (not exhaustive, needs design):
- A =kernels.conf= comment convention marking each entry with
  EOL date → simple grep to find stale entries
- A script that checks distro release feeds (RSS, release-monitoring.org,
  endoflife.date API) and flags new releases
- =download.sh --check-urls= to verify existing URLs still resolve
- Document the "add a new kernel" procedure so it's repeatable
- Periodic review cadence (e.g., at each quotatool release)

This is critical infrastructure — the test matrix IS the project's
regression safety net. A stale matrix means untested kernels in
production.

/Dependencies/: 4.4 (matrix must work first)

* Phase 8: Host-Independent Test Rootfs [0/5]

The test framework runs host-compiled binaries (glibc) inside old
kernels. glibc enforces a minimum kernel version at runtime. As host
distros update glibc, old kernels become untestable: =FATAL: kernel
too old=. This defeats the purpose of legacy kernel coverage.

Discovered 2026-03-09 on Fedora 43 (glibc 2.41, floor = 4.4):
debian-7, debian-8, ubuntu-1404, centos7 all fail. centos7 is tier 1.
See =research/glibc-kernel-floor.org= for full analysis and options.

Goal: replace host-dependent rootfs with a musl-based rootfs that has
*no kernel version floor*. The musl-static busybox in the initramfs
already proves this works — extend the approach to all test tools.

Recommended path: Option C (hybrid) — static musl quotatool + Alpine
minirootfs for tools. Only new test dep: =musl-gcc= (one package).
