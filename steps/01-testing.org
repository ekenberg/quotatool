#+TITLE: M1: Multi-Kernel Test Framework — Step Plan
#+DATE: 2026-03-08

* Overview

Build a local-first, CI-agnostic test framework that tests quotatool
across 15-20+ Linux kernel versions (including vendor-patched kernels)
and two filesystem types (ext4, XFS).

Architecture:
#+begin_example
test/                          ← workspace (all test framework code)
  run-tests.sh                 ← entry point: "run one command, get results"
  lib/
    boot.sh                    ← kernel boot layer (virtme-ng / raw QEMU)
    fs-setup.sh                ← loopback filesystem create/mount/teardown
    tap.sh                     ← TAP output helpers
  tests/
    t-block-limits.sh          ← individual test scripts
    t-inode-limits.sh
    t-grace-period.sh
    ...
  kernels/
    kernels.conf               ← matrix definition (name, vmlinuz path, boot method)
    download.sh                ← fetch + extract vendor kernels
    initramfs/                 ← custom initramfs for legacy kernels
  results/                     ← output (gitignored)

.github/workflows/test.yml    ← thin CI wrapper (calls run-tests.sh)
.cirrus.yml                   ← FreeBSD testing
#+end_example

Background: =research/testing-infrastructure.org=,
            =research/multi-kernel-testing.org=,
            =research/quotatool-build-analysis.org=

Key finding: quotatool ships its own quota struct definitions
(=src/linux/linux_quota.h=, =xfs_quota.h=, =dqblk_old.h=, =dqblk_v0.h=)
and does runtime detection of the kernel's quota interface in
=src/linux/quota.c:kern_quota_format()=. *One binary works across all
kernel versions.* Build once, test everywhere. But the runtime detection
code is the critical fragile point — it was last updated around the 4.x
kernel era and likely needs fixes for modern kernels.

* Phase 2: Test Suite [0/5]

Write the actual test cases. Each test is a self-contained function

** TODO 2.1 Define test case list
Based on quotatool's features. Reference =research/PLAN.org= test
case table. Each test must be independently runnable.

Minimum set:
| ID   | Test                            | FS        | Category |
|------+---------------------------------+-----------+----------|
| B01  | Set user block soft limit       | ext4, XFS | basic    |
| B02  | Set user block hard limit       | ext4, XFS | basic    |
| B03  | Set group block soft/hard limit | ext4, XFS | basic    |
| I01  | Set user inode soft limit       | ext4, XFS | basic    |
| I02  | Set user inode hard limit       | ext4, XFS | basic    |
| U01  | Human-readable units (K,M,G,T)  | ext4      | parse    |
| U02  | Relative adjustment (+100M)     | ext4      | parse    |
| F01  | Raise-only flag (-R)            | ext4      | feature  |
| F02  | Grace period set (-t)           | ext4      | feature  |
| F03  | Grace period restart (-r)       | ext4      | feature  |
| F04  | Numeric UID (:12345)            | ext4      | feature  |
| O01  | Verbose output (-v)             | ext4      | output   |
| O02  | Dump/machine-readable (-d)      | ext4      | output   |
| E01  | No root → graceful failure      | ext4      | error    |
| E02  | Invalid arguments               | ext4      | error    |
| E03  | Non-existent mount point        | ext4      | error    |
| D01  | Format detection succeeds       | ext4      | detect   |
| D02  | Format detection succeeds       | XFS       | detect   |
| D03  | Get+set round-trip (set, get back, compare) | ext4, XFS | detect |

*Category "detect"*: These test =kern_quota_format()= — the runtime
format detection in =src/linux/quota.c=. This is the most fragile code
path and the primary reason for multi-kernel testing. It probes
=/proc/fs/quota=, =/proc/sys/fs/quota=, and falls back through
=Q_V0_GETSTATS= → =Q_OLD_GETSTATS= → =Q_OLD_GETQUOTA=. It contains
vendor-specific workarounds (e.g., RedHat 2.4.2-2) and has not been
updated since ~kernel 4.x. *Any kernel where detection fails = test
failure = investigation needed.* These tests are the early warning
system for the entire project.

/Verification/: test list reviewed, each test has clear pass criteria.

** TODO 3.4 Build legacy initramfs (may be eliminated by Q4)

* Open Questions

** DONE Q1: Static linking for quotatool
*Not needed.* Source analysis shows quotatool ships its own quota
headers and does runtime format detection. One normally-linked build
works across all kernel versions. See =research/quotatool-build-analysis.org=.

If task 3.4 (legacy initramfs) is needed, the initramfs tools
(busybox, repquota, e2fsprogs) may need static builds or bundled
libs, but quotatool itself can be shared into the VM via 9p or
virtiofs. musl-cc remains an option for the *tools*, not for quotatool.

Tool note: [[https://github.com/intoli/exodus][exodus]] (GitHub: intoli/exodus)
bundles dynamic binaries with their .so files. Unmaintained (dead
since 2021). Noted for reference only.

/Affects/: 3.4 (only if Q4 fails)

** TODO Q2: Quota-tools (repquota) in legacy VMs
Need =repquota= inside legacy VMs for test verification. Since
quota-tools link against kernel headers and libc, there's risk of
version-dependent behavior. Best approach: extract =repquota= from
the matching distro package for each kernel (same vendor, same era).
This is a one-time parallel job alongside kernel downloads (3.3).

Each agent downloading a kernel also extracts the matching =quota=
package from the same distro repo. E.g., CentOS 7 kernel → CentOS 7
quota RPM → extract =repquota= binary + libs.

/Affects/: 3.3, 3.4

** WAITING Q3: XFS on very old kernels
CentOS 7 (3.10) does support XFS quotas, but the interface may differ
in edge cases. *Strategy*: try it, handle failures when we hit them.
Skip XFS tests on specific kernels if needed (add a =skip_xfs= flag
to kernels.conf).

/Affects/: 3.1

** TODO Q4: virtme-ng --force-9p on 4.x kernels
=--force-9p= tells virtme-ng to use the 9p filesystem protocol
(available since kernel 2.6.14) instead of virtiofs (requires 5.4+)
for sharing the host filesystem into the VM. If this works on 4.x
kernels, we can use virtme-ng for the entire legacy tier and *skip
building a custom initramfs entirely* (eliminates task 3.4).

*Test early*: download one 4.x kernel, try
=vng --force-9p -r <vmlinuz-4.x> -- uname -r=. If it boots and
prints the version, we're good. If not, fall back to raw QEMU.

Also test on 3.10 (oldest in matrix). If 9p works on 3.x too,
the two-tier split may be unnecessary.

/Affects/: 1.2, 3.4 (potentially eliminates 3.4)
